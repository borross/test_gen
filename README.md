# TestGen v2.3 — Генератор итоговых тестов по ИБ (CLI + веб + Docker)

Скрипт собирает итоговый тест из одного или нескольких markdown-файлов с тестами.
Поддерживает три типа вопросов, перемешивание вариантов ответов, фильтрацию по
сложности, весовое распределение вопросов по темам, генерацию нескольких вариантов
контрольной и режим проверки базы вопросов. Автоматически формирует блок правильных
ответов — в том числе развёрнутых текстовых ответов на открытые вопросы.

---

## Что нового в v2.0

### Исправленные ошибки парсера

| Проблема в v1 | Как исправлено |
|---------------|----------------|
| Вопрос или тест со словом «ответ» в названии («Ответственность оператора…») обрывал разбор | Блок ответов теперь распознаётся только по строгим маркерам: `## ОТВЕТЫ — …`, `**Ответы: …`, `Ответы на открытые вопросы:` |
| Разделители `---` и посторонний текст склеивались в развёрнутые открытые ответы | Накопление многострочного ответа прерывается на `---`, заголовках и жирных подзаголовках |
| Пары вида «3-D модель», «уровни 4-C» в тексте вопросов ложно попадали в ключ ответов и молча перезаписывали его | Ключ сканируется только внутри блока ответов и до подраздела открытых ответов |
| Вопрос с кириллическими буквами вариантов (`А.` `В.` `С.` `Д.`) исчезал молча | Кириллические буквы вариантов нормализуются в латинские; вопрос без вариантов теперь даёт явную ошибку |
| Буква в ключе, которой нет среди вариантов (`1-D` при трёх вариантах), попадала в итоговый ключ | Проверка согласованности: такой вопрос исключается с ошибкой |

### Новые возможности

- **Веб-интерфейс** (`web_app.py` + `index.html`) — работа со скриптом без командной строки
- **`--validate`** — проверка базы вопросов с отчётом об ошибках и предупреждениях
- **`--variants N`** — генерация N вариантов контрольной для параллельных групп
- **`--no-shuffle-questions`** / **`--group-by-source`** — управление порядком вопросов
- **`--force`** — защита от случайной перезаписи выходного файла
- Обнаружение дублей: повторный файл в `-i`, дублирующиеся номера вопросов, ответы без вопросов в ключе
- Добор недостающих вопросов теперь распределяется **пропорционально** между темами, а не из первых по порядку
- Поддержка BOM (файлы из Windows-редакторов) и любых тире в ключе (`1-B`, `1–B`, `1—B`)
- Юнит-тесты парсера (`test_test_gen.py`)

## Веб-интерфейс

Локальное веб-приложение поверх того же движка. Стандартная библиотека Python —
ни одного стороннего пакета, ни одного обращения в интернет: файлы с вопросами
не покидают ваш компьютер.

### Запуск

```bash
python web_app.py                 # http://127.0.0.1:8000, браузер откроется сам
python web_app.py --port 9000     # другой порт
python web_app.py --no-browser    # не открывать браузер автоматически
```

Файлы `web_app.py`, `index.html` и `test_gen.py` должны лежать в одной папке.

### Как устроена работа

1. **Загрузка** — перетащите `.md`-файлы в зону на главном экране (или выберите
   кнопкой). Один файл может содержать несколько тестов.
2. **База вопросов** — сразу после загрузки интерфейс показывает статистику
   (всего / базовых / обычных / открытых), детализацию по каждому тесту
   и отчёт проверки — те же ошибки и предупреждения, что даёт `--validate`.
   Вес каждого файла задаётся прямо в его строке.
3. **Параметры** — название, число вопросов, число вариантов, тип вопросов
   (пилюли показывают, сколько вопросов каждого типа доступно), порядок
   (случайный / по темам / как в источниках), перемешивание ответов, seed.
4. **Результат** — предпросмотр каждого варианта с составом по темам,
   скачивание `.md` по одному или всех сразу. Seed выводится на экран —
   сохраните его, чтобы воспроизвести тест позже (в интерфейсе или в CLI).

### API (если нужно автоматизировать)

| Метод | Путь | Назначение |
|-------|------|-----------|
| `POST` | `/api/parse` | `{"files":[{"name","content"}]}` → статистика базы, доступные типы, отчёт о проблемах |
| `POST` | `/api/generate` | те же `files` + `total`, `type`, `variants`, `weights`, `seed`, `shuffle`, `order` (`random`/`grouped`/`source`), `title`, `show_source` → варианты теста в markdown |

Ошибки параметров возвращаются кодом `400` с полем `error` на русском языке.

### Дизайн

Интерфейс построен на дизайн-системе Revolut: чёрный hero-блок с крупным
дисплейным заголовком, белая рабочая полоса с карточками (радиус 20px, без
теней), pill-кнопки высотой 48px, фирменный кобальт `#494fdf` — только на
одной акцентной карточке «Доступно для генерации» и точке вордмарка.
Адаптивная вёрстка до мобильных, видимый фокус с клавиатуры, поддержка
`prefers-reduced-motion`.

## Разворачивание в Docker

Если не хочется устанавливать Python — приложение разворачивается в Docker
одной командой. Образ на базе `python:3.12-slim`, работает от непривилегированного
пользователя, файловая система контейнера только для чтения, встроенный healthcheck.

### Быстрый старт

**Linux / macOS:**
```bash
git clone https://github.com/borross/test_gen
chmod +x deploy.sh
./deploy.sh up          # собрать образ и запустить → http://127.0.0.1:8000
```

**Windows (Docker Desktop):**
```bat
deploy.bat              :: то же самое, браузер откроется сам
```

**Docker Compose:**
```bash
docker compose up -d
```

### Команды deploy.sh / deploy.bat

| Команда | Действие |
|---------|----------|
| `up` | Собрать образ и запустить контейнер (по умолчанию) |
| `down` | Остановить и удалить контейнер |
| `rebuild` | Пересобрать образ без кэша и перезапустить (только `.sh`) |
| `logs` | Показать логи контейнера (следит в реальном времени) |
| `status` | Состояние контейнера и результат healthcheck |

### Настройка порта и доступа

```bash
TESTGEN_PORT=9000 ./deploy.sh up             # другой порт
TESTGEN_BIND=127.0.0.1 ./deploy.sh up        # доступ только с этой машины
```

Windows: `set TESTGEN_PORT=9000` / `set TESTGEN_BIND=127.0.0.1` перед `deploy.bat`.
Compose: те же переменные перед `docker compose up -d`.

### Доступ из сети и безопасность

- По умолчанию порт публикуется на всех интерфейсах (`0.0.0.0`) — интерфейс
  доступен и на самой машине (`http://127.0.0.1:8000`), и с других устройств
  вашей сети по адресу `http://<IP-машины>:8000`. Скрипт печатает оба адреса
  после запуска. Это важно, когда Docker работает на сервере или в ВМ.
- Если доступ из сети не нужен, ограничьте его: `TESTGEN_BIND=127.0.0.1`.
- Если интерфейс всё равно не открывается с другого устройства, проверьте
  файрвол хоста (Windows Defender / ufw) — порт должен быть разрешён.
- `--read-only` + `no-new-privileges` — контейнер не может писать в свою ФС
  и повышать привилегии (приложению это и не нужно: файлы обрабатываются в памяти).
- Скрипт перед запуском проверяет наличие Docker, всех файлов проекта
  и занятость порта — с понятной подсказкой при каждой проблеме.

### Ручная сборка (без скриптов)

```bash
docker build -t testgen:2.3 .
docker run -d --name testgen -p 8000:8000 testgen:2.3
```

## Типы вопросов

| Маркер | Тип | Описание |
|--------|-----|----------|
| `> ℹ️ Базовый` | Базовый тестовый | Вопрос с вариантами A/B/C/D, базовый уровень |
| *(без маркера)* | Обычный тестовый | Вопрос с вариантами A/B/C/D, стандартный уровень |
| `> ℹ️ Открытый` | Открытый | Вопрос без вариантов, требует развёрнутого ответа |

## Формат markdown-теста

```markdown
# 15 Безопасность приложений (Всего вопросов: 21, ...)

> ℹ️ Базовый
### 1. В чём принципиальная разница между багом и уязвимостью?
A. Это синонимы
B. Уязвимость — это баг с последствиями для CIA
C. Уязвимость возникает только в опенсорсе
D. Уязвимостями называют только сетевые проблемы

---

### 2. Что означает принцип Shift Left в DevSecOps?
A. Переход разработки на Linux
B. Встраивание проверок безопасности как можно раньше в жизненный цикл
C. Передача кода аудиторам только после релиза в прод
D. Расположение элементов IDE в левой части экрана

---

> ℹ️ Открытый
### 3. Что такое SQL-инъекция и в чём её опасность?

---

## ОТВЕТЫ — 15 Безопасность приложений
1-B, 2-B

**Ответы на открытые вопросы:**

3. SQL-инъекция — уязвимость, при которой злоумышленник внедряет произвольные
SQL-команды в пользовательский ввод, что может привести к утечке данных...
```

### Правила разметки

| Элемент | Формат |
|---------|--------|
| Заголовок теста | `# Название` или `## Название` — заголовок 1–2 уровня |
| Аннотация в заголовке | `(Всего вопросов: 21, ...)` или `(15 вопросов)` — автоматически убирается из названия |
| Маркер базового вопроса | `> ℹ️ Базовый` на строке перед `### N.` |
| Маркер открытого вопроса | `> ℹ️ Открытый` на строке перед `### N.` |
| Заголовок вопроса | `### N. Текст вопроса` |
| Варианты ответов | `A. текст` … `D. текст` (кириллические `А. В. С. Д.` тоже принимаются) |
| Начало блока ответов | `## ОТВЕТЫ — Название`, `ОТВЕТЫ:` или `**Ответы: …` — обязательно двоеточие или тире после слова |
| Пары ключа | `1-B, 2-C, …` — работает любой тип тире (`-`, `–`, `—`) |
| Блок открытых ответов | Строка `**Ответы на открытые вопросы:**`, затем `N. Текст ответа…` (многострочный текст допустим до ближайшего `---` или заголовка) |

> **Совместимость и нормализация:**
> - Кириллические двойники латинских букв (`А В С Д`) автоматически приводятся
>   к латинским — и в ключе ответов, и в буквах вариантов.
> - Файлы с BOM (сохранённые в Windows) читаются корректно.
> - Слово «ответ» внутри текста вопроса или названия теста («Ответственность…»)
>   разбор **не ломает** — блоком ответов считается только явный маркер.

## Аргументы командной строки

```
python test_gen.py -i FILE [FILE ...] [опции]
```

### Обязательные

| Аргумент | Описание |
|----------|----------|
| `-i`, `--input FILE [FILE ...]` | Один или несколько входных markdown-файлов. Дубли путей отбрасываются с предупреждением |

### Параметры генерации

| Аргумент | По умолчанию | Описание |
|----------|-------------|----------|
| `-n`, `--total N` | — | Количество вопросов в каждом варианте теста. Обязателен без `--info` / `--validate` |
| `-w`, `--weights W [W ...]` | равные веса | Веса для каждого **файла** (неотрицательные, сумма > 0). Если файл содержит N тестов, его вес делится поровну между ними |
| `--type TYPE` | `mix` | Тип вопросов (см. таблицу ниже) |
| `--shuffle` | выкл. | Перемешать варианты ответов внутри каждого тестового вопроса. Правильная буква пересчитывается автоматически |
| `--variants N` | `1` | Сгенерировать N независимых вариантов теста. Файлы получают суффиксы `_v1`, `_v2`, … |
| `--no-shuffle-questions` | выкл. | Не перемешивать порядок вопросов — сохранить порядок тем и номеров из источников |
| `--group-by-source` | выкл. | Сгруппировать вопросы по тестам-источникам (внутри темы порядок случайный) |

### Типы вопросов (`--type`)

| Значение | Что включается в тест |
|----------|-----------------------|
| `mix` | Все тестовые вопросы (базовые + обычные) — **по умолчанию** |
| `basic` | Только базовые тестовые (`> ℹ️ Базовый`) |
| `normal` | Только обычные тестовые (без маркера) |
| `open` | Только открытые (`> ℹ️ Открытый`) |
| `all` | Все вопросы: тестовые + открытые |

### Параметры вывода

| Аргумент | По умолчанию | Описание |
|----------|-------------|----------|
| `-o`, `--output FILE` | `generated_test.md` | Имя выходного файла. При `--variants N` — база имени для `_v1…_vN` |
| `--force` | выкл. | Перезаписать существующий выходной файл. Без флага скрипт откажется затирать файл |
| `--title TITLE` | `Итоговый тест по ИБ` | Название теста. При нескольких вариантах добавляется « — Вариант N» |
| `--show-source` | выкл. | Пометка под каждым вопросом: тест-источник и оригинальный номер |

### Утилиты

| Аргумент | Описание |
|----------|----------|
| `--info` | Статистика по всем найденным тестам; генерация не выполняется |
| `--validate` | Проверка базы: битые вопросы, ответы без вопросов, дубли номеров, несогласованный ключ. Код выхода `1` при ошибках — удобно для CI |
| `--seed N` | Фиксированный seed. Одинаковый seed + входные данные → идентичный результат. Без флага seed генерируется и печатается — его можно переиспользовать для воспроизведения |

## Примеры использования

### Проверить базу вопросов перед генерацией

```bash
python test_gen.py -i all_tests.md --validate
```

Вывод:
```
🔍 Результат проверки: 2 ошибок, 1 предупреждений

Ошибки (вопросы исключены из базы):
  ❌  [all_tests.md / '16 Ответственность в сфере ИБ'] Вопрос 2: в ключе указан
      ответ «D», но такого варианта нет (есть: A, B) — пропускаем
  ❌  [all_tests.md / '16 Ответственность в сфере ИБ'] Вопрос 3: не найдено
      ни одного варианта ответа — пропускаем (проверьте формат «A. текст»)

Предупреждения:
  ⚠  [all_tests.md / '16 Ответственность в сфере ИБ'] В ключе есть ответы
      без вопросов: 9
```

### Посмотреть состав файла

```bash
python test_gen.py -i all_tests.md --info
```

### Обычный тест (по умолчанию — только тестовые вопросы)

```bash
python test_gen.py -i all_tests.md -n 20 --shuffle
```

### Четыре варианта контрольной для параллельных групп

```bash
python test_gen.py -i all_tests.md -n 20 --shuffle --variants 4 \
  --seed 42 --title "Контрольная №2" -o kontr2.md
```

Создаёт `kontr2_v1.md` … `kontr2_v4.md`. Каждый вариант — независимая выборка
вопросов со своим перемешиванием, но всё воспроизводимо от одного seed.

### Только базовые / только открытые вопросы

```bash
python test_gen.py -i all_tests.md -n 10 --type basic --shuffle
python test_gen.py -i all_tests.md -n 6  --type open
```

### Смешанный тест с группировкой по темам

```bash
python test_gen.py -i all_tests.md -n 15 --type all --shuffle --group-by-source
```

### Тест в порядке источников (для разбора на занятии)

```bash
python test_gen.py -i all_tests.md -n 15 --no-shuffle-questions
```

### Несколько файлов с весами

```bash
python test_gen.py -i all_tests.md extra.md -w 70 30 -n 20 --shuffle
```

70% вопросов из `all_tests.md`, 30% из `extra.md`.
Если `all_tests.md` содержит 3 теста, его вес `70` делится: по `23.3` на тест.
Если в какой-то теме вопросов не хватает, недостающие добираются
**пропорционально** из остальных тем.

### Полный пример

```bash
python test_gen.py \
  -i all_tests.md extra_tests.md \
  -w 60 40 \
  -n 25 \
  --type all \
  --shuffle \
  --variants 2 \
  --title "Итоговый экзамен по ИБ" \
  -o exam_final.md \
  --seed 2026 \
  --force
```


## Логика весов при нескольких тестах в файле

Веса задаются **на файл**, а не на отдельный тест внутри него:

```
Файл A (вес 60) содержит 3 теста  →  каждый тест получает вес 60 / 3 = 20
Файл B (вес 40) содержит 1 тест   →  тест получает вес 40
```

Итого 4 теста с весами 20 : 20 : 20 : 40.
Недостающие вопросы (если в какой-то теме их меньше расчётного) добираются
из остальных тем пропорционально их остаточной ёмкости.

## Обработка ошибок и граничные случаи

| Ситуация | Поведение |
|----------|-----------|
| Запрошено больше вопросов, чем есть в базе | Предупреждение; тест с максимально доступным количеством |
| Тестовый вопрос без буквенного ответа в ключе | **Ошибка**; вопрос пропускается |
| Буква в ключе, которой нет среди вариантов | **Ошибка**; вопрос пропускается |
| Вопрос без единого распознанного варианта | **Ошибка** с подсказкой формата; вопрос пропускается |
| Дублирующийся номер вопроса внутри теста | **Ошибка**; повтор пропускается |
| Ответы в ключе без соответствующих вопросов | Предупреждение с номерами |
| Открытый вопрос без текстового ответа | Предупреждение; включается с пометкой «ответ не указан» |
| Разные буквы для одного номера в ключе | **Ошибка**; остаётся первая |
| Один файл указан в `-i` дважды | Предупреждение; учитывается один раз |
| Выходной файл уже существует | Отказ с подсказкой про `--force` |
| Веса: количество ≠ файлам, отрицательные, сумма 0 | Ошибка аргументов |
| Кириллица `А В С Д` в ключе и в вариантах | Нормализуется в латиницу |
| BOM в начале файла, тире `–`/`—` в ключе | Обрабатываются прозрачно |
| Слово «ответ» в тексте вопроса или названии теста | Разбор не ломается |
